# Services

Business logic modules. IPC handlers delegate to these — they contain no business logic themselves.
Source: `src/main/services/`

| Module | Purpose | Key Exports |
|--------|---------|-------------|
| `repo-search-service.ts` | Grep-based codebase search for workbench research — literal fixed-string search to prevent ReDoS | `searchRepo`, `parseGrepOutput`, `RepoSearchResult`, `RepoSearchMatch` |
| `adhoc-promotion-service.ts` | Validates and promotes a completed adhoc agent worktree into the Code Review queue as a sprint task | `promoteAdhocToTask`, `PromoteAdhocParams`, `PromoteAdhocResult` |
| `webhook-delivery-service.ts` | Constructs and delivers test webhook events including HMAC-SHA256 signing and HTTP POST delivery | `deliverWebhookTestEvent`, `WebhookTestResult` |
| `checkpoint-service.ts` | Creates a git commit snapshot of the current agent worktree state without stopping the agent | `createCheckpoint`, `CheckpointResult` |
| `epic-group-service.ts` | Owns the in-memory epic dependency index and the single entry point for all epic mutations; exports typed `EpicNotFoundError` / `EpicCycleError` so adapters can map to MCP / IPC error codes without regex-matching | `EpicGroupService`, `createEpicGroupService`, `EpicNotFoundError`, `EpicCycleError` |
| `planner-mcp-server.ts` | In-process SDK MCP server exposing BDE's task and epic CRUD to adhoc + assistant agents. Tools: `tasks.create`, `tasks.update`, `tasks.list`, `epics.create`, `epics.list`, `epics.addTask`, `epics.setDependencies`, `meta.repos`, `meta.taskStatuses`. Every write routes through `sprint-service` / `EpicGroupService` — same validation, audit trail, dependency auto-blocking, and renderer broadcast as the UI. **T-19 (Phase 2):** moved out of `agent-manager/` because its sole responsibility is service orchestration. | `createPlannerMcpServer`, `buildPlannerTools`, `PLANNER_MCP_SERVER_NAME`, `PlannerMcpDeps`, `PlannerTool` |
| `cost-queries.ts` (`src/main/`) | Thin wrapper delegating to `data/cost-queries`. All functions accept optional `db?: Database.Database` for injection. | `getCostSummary`, `getRecentAgentRunsWithCost`, `getAgentHistory` |
| `settings.ts` (`src/main/`) | Facade over `data/settings-queries` that transparently encrypts/decrypts sensitive keys (`SENSITIVE_SETTING_KEYS`) via `secure-storage.ts`. Lazy re-encryption of legacy plaintext values on first read (skipped with a warning if encryption unavailable). `setSetting()` will throw if encryption is unavailable for sensitive keys. All functions accept optional `db?: Database.Database` for injection. | `getSetting`, `setSetting`, `deleteSetting`, `getSettingJson`, `setSettingJson` |
| `agent-history.ts` (`src/main/`) | Agent run metadata and log management — persistent storage in SQLite agent_runs table. All exported functions accept optional `db?: Database.Database` for injection. | `listAgents`, `createAgentRecord`, `appendLog`, `readLog`, `getAgentMeta`, `updateAgentMeta`, `pruneOldAgents`, `hasAgent`, `findAgentByPid`, `listAgentRunsByTaskId`, `finalizeStaleAgentRuns`, `reconcileRunningAgentRuns`, `finalizeAllRunningAgentRuns`, `backfillUtcTimestamps` |
| `adhoc-agent.ts` (`src/main/`) | Ad-hoc / assistant agent spawning. Now passes `disallowedTools: [...PIPELINE_DISALLOWED_TOOLS]` in `baseOptions` so adhoc sessions apply the same network/exfiltration restrictions as pipeline agents (blocks `curl`, `wget`, `WebFetch`, `WebSearch`, `gh`, git-recon commands). | `spawnAdhocAgent`, `getAdhocHandle`, `ImageAttachment` |
| `review-service.ts` | Orchestrates reviewer SDK calls, diff fetching, result caching, and result aggregation for auto-review. Resolves the reviewer model per-call via an injected `resolveAgentRuntime` dep (bound to `agents.backendConfig.reviewer` by the composition root). Auto-review `runSdkOnce` call explicitly opts into `permissionMode: 'bypassPermissions'` with `tools: []` — the tool list is empty so bypass is a no-op, but the explicit opt-in is required after the `sdk-streaming.ts` hardening removed the silent bypass default. | `ReviewService`, `ReviewServiceDeps`, `createReviewService`, `WorktreeMissingError` |
| `spec-synthesizer.ts` | Codebase-aware spec generation (`synthesizeSpec`) and revision (`reviseSpec`) for the Task Workbench. Gathers file tree + keyword-matched code snippets, then streams SDK output with `tools: []` + explicit `bypassPermissions` opt-in. Resolves the synthesizer model via `resolveAgentRuntime('synthesizer')`. | `synthesizeSpec`, `reviseSpec`, `cancelSynthesis`, `SynthesizeResult` |
| `spec-generation-service.ts` | Quick-spec generator from title + repo + template hint (used by the Task Workbench "Generate Spec" button). Single-turn SDK call via internal `runSdkPrint` with `tools: []` + explicit `bypassPermissions` opt-in. Resolves the synthesizer model via `resolveAgentRuntime('synthesizer')`. | `buildSpecGenerationPrompt`, `generateSpec` |
| `copilot-service.ts` | Builds the Task Workbench copilot's chat prompt and the restricted SDK options for every copilot invocation (read-only tool list, dollar ceiling, turn limit, resolved model). `getCopilotSdkOptions(repoPath, model, extras?)` takes the routed model as a required second positional argument and explicitly opts into `permissionMode: 'bypassPermissions'` / `allowDangerouslySkipPermissions: true` — bypass is no longer a silent default in `sdk-streaming.ts`, so every caller with a vetted narrow tool list opts in here. | `COPILOT_ALLOWED_TOOLS`, `COPILOT_DISALLOWED_TOOLS`, `COPILOT_MAX_BUDGET_USD`, `COPILOT_MAX_TURNS`, `buildChatPrompt`, `getCopilotSdkOptions` |
| `review-ship-batch.ts` | Batch Ship It orchestration — merges each selected review task onto local `main` in order (reusing the single-task shipIt plan minus the per-task `push`) and issues a single terminal `git push`. Aborts on the first per-task failure and returns `{ failedTaskId, shippedTaskIds }` so the caller can report which task failed and which merges are sitting on local main. Requires all tasks to share one repo. | `shipBatch` |
| `review-orchestration-service.ts` | Thin facade over `review-action-policy` + `review-action-executor`. `createPr` is the exception — it has its own inline flow: pushes the branch, calls `createPullRequest`, updates `pr_url`/`pr_number`/`pr_status='open'`, cleans up the worktree, then sets `worktree_path: null` and returns. **The task intentionally stays in `review` status** — the sprint PR poller watches `pr_status='open'` tasks and transitions to `done` (or `cancelled`) when GitHub reports the PR as merged (or closed without merge). | `mergeLocally`, `createPr`, `requestRevision`, `discard`, `shipIt`, `rebase` |
| `review-action-policy.ts` | Pure policy layer — maps a `ReviewActionInput` to a `ReviewActionPlan` (gitOps list + taskPatch + terminalStatus). Each action has one dedicated builder. `buildCreatePrPlan` sets `taskPatch: { worktree_path: null }` and `terminalStatus: null` to match the actual orchestration service behaviour (PR creation does not mark the task done). | `classifyReviewAction`, `ReviewActionInput`, `ReviewActionPlan`, `GitOpDescriptor` |
| `review-orchestration-types.ts` | Input/result type definitions for the review orchestration service. Adds `ShipBatchInput` + `ShipBatchResult` (discriminated union: success reports `shippedTaskIds`; failure includes `failedTaskId` + merged-so-far list). | `MergeLocallyInput`, `MergeLocallyResult`, `CreatePrInput`, `CreatePrResult`, `RequestRevisionInput`, `RequestRevisionResult`, `DiscardInput`, `DiscardResult`, `ShipItInput`, `ShipItResult`, `ShipBatchInput`, `ShipBatchResult`, `RebaseInput`, `RebaseResult` |
| `review-response-parser.ts` | Parses and validates raw reviewer model output into a typed `ParsedReview` shape | `parseReviewResponse`, `MalformedReviewError`, `ParsedReview` |
| `batch-import.ts` | Validates and bulk-creates sprint tasks from an import payload. Validates status values against `TASK_STATUSES` and repo names against the configured repo list. | `batchImportTasks`, `BatchImportResult` |
| `workflow-engine.ts` | Instantiates a workflow template as a set of linked sprint tasks | `instantiateWorkflow` |
| `sprint-mutation-broadcaster.ts` | Notification orchestration for sprint task mutations — in-process listeners, debounced `sprint:externalChange` IPC broadcast (200ms), and webhook dispatch | `notifySprintMutation`, `onSprintMutation`, `SprintMutationEvent`, `SprintMutationListener` |
| `sprint-service.ts` | Backward-compatible façade over `sprint-mutations` + `sprint-mutation-broadcaster`. Wraps every mutation to auto-trigger notifications. Exports `createTaskWithValidation` — single entry point for task creation used by both the IPC handler and the MCP server: runs structural validation, required-sections check, repo-configured check, then delegates to `createTask`. Accepts an optional `{ skipReadinessCheck }` flag that bypasses spec-structure checks for batch/admin flows while still enforcing title/repo and repo configuration (logged at warn level when true). Failures throw `TaskValidationError` with a machine-readable `code` (`'spec-structural'` \| `'spec-readiness'` \| `'repo-not-configured'`) so callers can branch without parsing the message. `updateTask(id, patch, options?)` forwards `options.caller` to the data layer so MCP-originated writes land as `'mcp'` in `task_changes.changed_by` (instead of `'unknown'`). `cancelTask(id, { reason?, caller? }, deps)` forwards the same attribution, catches the data-layer "Invalid transition" throw and re-raises it as `TaskTransitionError` (carrying `taskId`, `fromStatus`, `toStatus`) so adapters can translate to structured error kinds without regex-matching the underlying message; unknown errors propagate unchanged. Also exports `resetTaskForRetry(id, deps?)` — clears stale terminal-state fields (`completed_at`, `failure_reason`, `claimed_by`, `started_at`, `retry_count`, `fast_fail_count`, `next_eligible_at`) so a re-queued task looks fresh; does not set `status` (caller owns that). Re-exports `getTaskChanges` (+ `TaskChange`, `GetTaskChangesOptions`) from `data/task-changes` so adapters like the MCP server consume audit history through the service layer instead of reaching into the data layer. | `createTask`, `updateTask`, `deleteTask`, `claimTask`, `releaseTask`, `forceUpdateTask`, `createReviewTaskFromAdhoc`, `cancelTask`, `createTaskWithValidation`, `resetTaskForRetry`, `getTaskChanges`, `CancelTaskDeps`, `CancelTaskOptions`, `CreateTaskWithValidationDeps`, `CreateTaskWithValidationOpts`, `ResetTaskForRetryDeps`, `TaskTransitionError`, `TaskValidationError`, `TaskValidationCode`, `TaskChange`, `GetTaskChangesOptions`, `UpdateTaskOptions`, re-exports from `sprint-mutations` and `sprint-mutation-broadcaster` |
| `spec-quality/index.ts` | Public API for the spec-quality service. Re-exports `createSpecQualityService` and exposes `validateTaskSpec` — the single entry point handlers and services use to run structural + semantic checks on a task spec before `queue`/`unblock` transitions. Throws with a context-prefixed error on failure. | `createSpecQualityService`, `validateTaskSpec` |
| `spec-quality/validators/sync-validators.ts` | Synchronous spec validators — `SizeWarningsValidator` blocks specs >1000 words (error) and warns at >500; `RequiredSectionsValidator`, `FilePathsValidator`, `NumberedStepsValidator`, `BannedPhrasesValidator` | `SizeWarningsValidator`, `RequiredSectionsValidator`, `FilePathsValidator`, `NumberedStepsValidator`, `BannedPhrasesValidator` |
| `spec-quality/validators/prescriptiveness-validator.ts` | Async AI-based check for exploration language in Implementation Steps — returns error (not warning) when design decisions detected; returns `PRESCRIPTIVENESS_CHECK_FAILED` warning on SDK failure instead of silently passing | `PrescriptivenessValidator` |
| `task-terminal-service.ts` | Unified handler for all task terminal-status paths — records metrics, schedules dependency resolution (via `scheduleDependencyResolution`), and schedules worktree cleanup. All terminal paths converge here. Accepts optional `runInTransaction` dep; when provided (wired via `index.ts`), wraps the full cascade-cancellation loop in a single SQLite transaction for atomicity. | `TaskTerminalService`, `createTaskTerminalService`, `TaskTerminalServiceDeps` |
| `pr-poller.ts` (`src/main/`) | Polls GitHub for open PRs on all configured repos every 60s, fetches check-run summaries, and broadcasts `pr:listUpdated`. Uses `Promise.allSettled` so one failing repo does not abort the rest. Includes exponential backoff on poll errors. | `startPrPoller`, `stopPrPoller`, `getLatestPrList`, `refreshPrList`, `POLL_INTERVAL_MS` |
| `sprint-pr-poller.ts` (`src/main/`) | Polls PR statuses for sprint tasks with `pr_status=open` every 60s and auto-transitions tasks when PRs merge or close. DI-based via `createSprintPrPoller`; legacy `startSprintPrPoller` wraps it. | `createSprintPrPoller`, `startSprintPrPoller`, `stopSprintPrPoller`, `SprintPrPollerDeps`, `SprintPrPollerInstance` |
| `github-pr-status.ts` (`src/main/`) | Fetches individual PR status (merged/state/mergeableState) via GitHub REST API. `pollPrStatuses` uses `Promise.allSettled` so one failing PR fetch does not abort the rest. | `pollPrStatuses`, `PrStatusInput`, `PrStatusResult` |
| `bootstrap.ts` (`src/main/`) | App startup utilities — DB initialization, file watcher, CSP, cleanup tasks, PR pollers, background services. Tracks non-trivial async startup errors and surfaces them to the renderer via `emitStartupWarnings()` (call at `ready-to-show`). Trivial conditions (missing credentials, already-skipped imports) are filtered out. `validateRepoPaths()` checks each configured repo's `localPath` with `existsSync` at startup and queues a warning toast for any paths that don't exist on this machine. `warnPlaintextSensitiveSettings()` proactively re-encrypts any sensitive settings (github.token, supabase.serviceKey) found without the `ENC:` prefix — no longer just warns, now fixes silently or warns only on failure. | `initializeDatabase`, `startDbWatcher`, `setupCSP`, `setupCleanupTasks`, `startPrPollers`, `startBackgroundServices`, `emitStartupWarnings`, `warnPlaintextSensitiveSettings`, `validateRepoPaths`, `DEBOUNCE_MS` |
| [`credential-service.ts`](./credential-service.md) | Unified Claude + GitHub credential resolution with discriminated-return shape. Primary entry point for credential checks. `oauth-checker.checkOAuthToken` and `auth-guard.ensureSubscriptionAuth` now delegate here so every call site produces the canonical `CREDENTIAL_GUIDANCE` actionable message. Caches success 5 min, failure 30 s. Used by `spawn-and-wire.ts` for pre-spawn refresh (V0.6), `pr-operations.ts` for the `gh` auth guard, and the drain-loop precondition. | `createCredentialService`, `getDefaultCredentialService`, `CredentialService`, `CredentialResult`, `CredentialKind`, `CREDENTIAL_GUIDANCE`, `ClaudeCredentialStore`, `GithubCredentialStore` |
| `credential-store.ts` (`src/main/`) | Low-level Claude credential I/O — reads the macOS Keychain (`/usr/bin/security`) and the file-based OAuth token fallback, detects the Claude Code CLI on disk, and reports a unified `AuthStatus`. Rate-limited to one Keychain read per second with a cached result. Shared primitive imported by `auth-guard.ts` and `credential-service.ts` so those two don't depend on each other. | `checkAuthStatus`, `CredentialStore`, `MacOSCredentialStore`, `defaultCredentialStore`, `AuthStatus`, `KeychainOAuth`, `KeychainPayload` |
| `auth-guard.ts` (`src/main/`) | High-level subscription-auth gate. `ensureSubscriptionAuth` clears `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` env vars then delegates to `credential-service.getCredential('claude')` for the canonical guidance message. Tests that inject a custom `CredentialStore` take a direct `checkAuthStatus` path. | `ensureSubscriptionAuth` |
| `git.ts` (`src/main/`) | Thin `execFile` adapter around the git CLI — `status`, `diff`, `stage`, `unstage`, `commit`, `push`, `branches`, `checkout`, `fetch`, `pull`, plus `detectGitRemote` (parses `origin` URL via `parseGitHubRemote`). Repo path lookup via `getRepoPaths` / `getRepoPath` (case-insensitive). Write ops return a structured `Result` so handlers can surface errors without parsing shell output. | `getRepoPaths`, `getRepoPath`, `gitStatus`, `detectGitRemote`, `gitDiffFile`, `gitStage`, `gitUnstage`, `gitCommit`, `gitPush`, `gitBranches`, `gitCheckout`, `gitFetch`, `gitPull`, `GitFileStatus` |
| `epic-group-service.ts` | Sole owner of the in-memory epic dependency graph. Mutation-side facade over `task-group-queries` plus the read methods (`getDependentEpics`, `areEpicDepsSatisfied`) that `TaskTerminalService` and `AgentManager` consume as an `EpicDepsReader`. Rebuilds the graph on every mutation; cycle detection via `detectEpicCycle` rejects circular deps atomically. Throws `EpicNotFoundError` on missing-epic lookups and `EpicCycleError` (carrying the cycle path) on cycle violations so MCP/IPC adapters can branch on type. `createEpic` / `updateEpic` carry tri-state `goal` semantics from the MCP schema all the way through to SQL: a `null` `goal` clears the column (stored as SQL NULL); an omitted (or explicitly `undefined`) `goal` leaves the column untouched — no silent coercion. | `createEpicGroupService`, `EpicGroupService`, `EpicGroupQueries`, `EpicNotFoundError`, `EpicCycleError` |
| `status-server.ts` | Read-only HTTP status server. Accepts optional `token` parameter; when provided, enforces `Authorization: Bearer <token>` on every request. Validates `Host` header against `127.0.0.1:<boundPort>` / `localhost:<boundPort>` to prevent DNS-rebinding attacks. `boundPort` is tracked post-bind so port=0 (random, used in tests) works correctly. | `createStatusServer`, `StatusServer` |
| `logger.ts` (`src/main/`) | File-backed logger that writes to `~/.bde/bde.log` with rotation at 10 MB (keeps 3 generations). Enforces 0600 permissions on `bde.log` on every `createLogger` call and `mode: 0o600` on every `appendFileSync`, so the log — which may contain agent prompts, task specs, and SDK errors that echo tokens — is not world-readable under the default umask. Mirror writes to `console.*` are gated by `CONSOLE_LOG_ENABLED` — off in packaged builds (`NODE_ENV === 'production'`) unless the user sets `BDE_CONSOLE_LOG=1`. | `createLogger`, `Logger`, `logError` |
| `db.ts` (`src/main/`) | SQLite connection factory, WAL mode, and migration runner. `backupDatabase()` performs `VACUUM INTO` to `bde.db.backup` and then `chmodSync(backupPath, 0o600)` — the backup contains the same secrets (encrypted tokens, HMAC keys, settings) and must not inherit the user umask. | `getDb`, `closeDb`, `backupDatabase`, `runMigrations`, `migrations` |
| `pty.ts` (`src/main/`) | Lazy-loaded PTY wrapper for the IDE's integrated terminal. Spawns with `buildAgentEnv()` scrubbed env (not full `process.env`) so AWS keys, NPM tokens, ANTHROPIC_API_KEY and other shell-exported secrets cannot be read via `env` in the integrated terminal. `TERM=xterm-256color` is re-added on top of the scrubbed env. | `isPtyAvailable`, `validateShell`, `createPty`, `PtyHandle`, `_setPty` |
| `fs.ts` (`src/main/`) | Memory filesystem + open-dialog IPC handlers. `openDirectoryDialog` invokes `rememberApprovedIdeRoot` on user acceptance so the new default-deny `validateIdeRoot` guard in `ide-fs-handlers.ts` will accept the chosen folder. | `registerFsHandlers`, `listMemoryFiles`, `readMemoryFile`, `writeMemoryFile`, `validateMemoryPath` |
| `index.ts` (`src/main/`) | Electron main entry. `createWindow()` wraps `new BrowserWindow(…)` in try/catch and shows an error dialog + `app.quit()` on failure, preventing the silent hang that used to happen on headless systems or when Electron could not initialize a graphics context. | `createWindow` (module-private; no runtime exports) |
